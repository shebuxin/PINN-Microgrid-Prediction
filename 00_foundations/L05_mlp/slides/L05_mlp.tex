\begin{frame}{MLP: stack nonlinear layers, get a curved boundary}
An MLP composes \emph{affine maps} with \emph{nonlinear activations}:
\begin{equation*}
\boxed{\;h_{l+1} \;=\; \sigma\!\left( W_l \, h_l + b_l \right), \quad l = 0, 1, \ldots, L-1\;}
\end{equation*}
\vspace{0.2em}

\draw[->, thick] (-0.3,0) -- (5.0,0) node[right]{$x_1$};

}
\foreach \pt in {(0.8,2.8), (1.2,3.0), (1.6,2.6), (0.6,2.5), (1.4,2.9), (1.0,2.4)} {

\begin{block}{Why the curved boundary is possible}
Each layer applies an affine map (so far still linear), \emph{then} a nonlinearity $\sigma(\cdot)$. Stacking these compositions builds curved decision regions. The capacity grows with depth and width.
\end{block}
\end{frame}

\begin{itemize}

\begin{frame}{Output heads for different tasks}
\centering
\small
\begin{tabular}{lll}
\toprule
Task & Output activation & Loss \\
\midrule
Binary classification (safe vs unsafe) & sigmoid & binary cross-entropy (BCE) \\
Multi-class classification (5 violation types) & softmax & cross-entropy \\
Regression (severity score) & identity / linear & mean squared error (MSE) \\
Ranking (top-k risk order) & identity & pairwise hinge / ranking loss \\
\textbf{Multi-task (project's PI-MLP)} & multiple heads at once & weighted sum of the above \\
\bottomrule
\end{tabular}
\vspace{0.6em}
\begin{block}{What ``multi-task'' looks like architecturally}
The shared layers extract a common feature representation. Then multiple heads branch off, each with its own loss. The total loss is $\mathcal{L} = \lambda_1 \mathcal{L}_{\text{cls}} + \lambda_2 \mathcal{L}_{\text{sev}} + \lambda_3 \mathcal{L}_{\text{rank}} + \ldots$. The project's Week 6 PI-MLP does exactly this.
\end{block}
\end{frame}

% ============================================================
\begin{frame}{Choosing width and depth}
\begin{columns}[T]
\column{0.5\textwidth}
\textbf{Rules of thumb}
\begin{itemize}
  \item Start small: 1--2 hidden layers, 32--128 units.
  \item Train. Look at train and validation loss.
  \item Underfitting (both losses high): widen or deepen.
  \item Overfitting (train low, val high): shrink or regularize.
\end{itemize}
\column{0.5\textwidth}
\textbf{The project's regime}
\begin{itemize}
  \item Only 77 labeled samples.
  \item Too much capacity $\to$ memorize the training set, generalize poorly.
  \item Week 5/6 use small MLPs ($\sim 32$ hidden units, 2 layers) with weight decay and early stopping.
\end{itemize}
\end{columns}
\vspace{0.5em}
\begin{alertblock}{Don't reach for a 10-layer transformer}
On a small tabular dataset like ours, simple is competitive. Modern research often finds that gradient boosting beats deep nets on small tabular data -- and that a well-tuned 2-hidden-layer MLP beats most things below 1000 samples.
\end{alertblock}
\end{frame}

\section*{3. Training}

\begin{frame}{The training loop in one slide}
\begin{enumerate}
  \item Pick a minibatch of data $(X, y)$ from the training set.
  \item \textbf{Forward pass}: compute predictions $\hat{y} = f_\theta(X)$.
  \item \textbf{Loss}: compute $\mathcal{L}(\hat{y}, y)$ -- a scalar measuring how wrong we are.
  \item \textbf{Backward pass}: compute $\nabla_\theta \mathcal{L}$ via the chain rule.
  \item \textbf{Update}: $\theta \leftarrow \theta - \eta \, \nabla_\theta \mathcal{L}$ (gradient descent step).
  \item Go to 1. Repeat for many epochs until validation loss stops improving.
\end{enumerate}
\vspace{0.6em}
\begin{block}{Three knobs in this loop}
\textbf{Batch size} (how many samples per step), \textbf{learning rate} $\eta$ (step size), and \textbf{number of epochs} (passes through the data). Tuning these is most of the practical art.
\end{block}
\end{frame}

% ============================================================
\begin{frame}{Common loss functions}
\textbf{Binary classification.} Output $\hat{p} \in [0,1]$ (sigmoid), target $y \in \{0,1\}$:
\begin{equation*}
\mathcal{L}_{\text{BCE}} \;=\; -\bigl[y \log \hat{p} + (1-y) \log(1-\hat{p})\bigr]
\end{equation*}
Penalizes confident-wrong predictions heavily.
\vspace{0.4em}

\textbf{Regression.} Output $\hat{y}$, target $y$:
\begin{equation*}
\mathcal{L}_{\text{MSE}} \;=\; (\hat{y} - y)^2
\end{equation*}
\textbf{Multi-class classification.} Output probabilities $\hat{p}_k$ over $K$ classes:
\begin{equation*}
\mathcal{L}_{\text{CE}} \;=\; -\sum_k y_k \log \hat{p}_k
\end{equation*}
\vspace{0.2em}
\begin{block}{Class-weighted BCE for imbalance}
\begin{equation*}
\mathcal{L}_{\text{wBCE}} = -\bigl[\,w_1 \cdot y \log \hat{p} + w_0 \cdot (1-y) \log(1-\hat{p})\bigr]
\end{equation*}
Set $w_1 > w_0$ if the positive (unsafe) class is rarer than the negative. The project uses this for screening recall.
\end{block}
\end{frame}

% ============================================================
\begin{frame}{Backpropagation in one slide}
The training step needs $\partial \mathcal{L}/\partial W_l$ for every layer. By the chain rule:
\begin{equation*}
\frac{\partial \mathcal{L}}{\partial W_l} \;=\; \frac{\partial \mathcal{L}}{\partial h_{L}} \cdot \frac{\partial h_{L}}{\partial h_{L-1}} \cdots \frac{\partial h_{l+1}}{\partial W_l}
\end{equation*}
\begin{itemize}
  \item Each factor is a small Jacobian computable in closed form for ReLU / sigmoid / tanh / linear layers.
  \item Multiplying them out from the output back toward the input is the ``backward pass''.
  \item \textbf{PyTorch's autograd does this for you.} You never derive gradients by hand.
\end{itemize}
\vspace{0.4em}
\begin{alertblock}{The one rule}
You write the forward pass. Calling \code{loss.backward()} fills in \code{param.grad} for every parameter. You then call \code{optimizer.step()}. That's it -- no manual calculus.
\end{alertblock}
\end{frame}

% ============================================================
\begin{frame}{Optimizers: SGD, momentum, Adam}
\centering
\small
\begin{tabular}{lll}
\toprule
Optimizer & Update rule (per parameter $\theta$) & When to use \\
\midrule
\textbf{SGD} & $\theta \leftarrow \theta - \eta \cdot \nabla \mathcal{L}$ & simple, well-tuned baselines \\
\textbf{SGD + momentum} & $v \leftarrow \mu v + \nabla \mathcal{L}$; $\theta \leftarrow \theta - \eta v$ & smoother trajectories \\
\textbf{Adam} & per-parameter adaptive $\eta$ using $1^{\text{st}}$ and $2^{\text{nd}}$ moment estimates & default choice today \\
\bottomrule
\end{tabular}
\vspace{0.6em}
\begin{block}{Recommended default for this course}
\code{torch.optim.Adam(model.parameters(), lr=1e-3)}. Works well on $\sim 90\%$ of small-to-medium problems. For very small datasets like ours, you may need to drop to $\text{lr} = 10^{-4}$ to avoid oscillation.
\end{block}
\end{frame}

\vspace{0.4em}
\begin{columns}[T]
\column{0.5\textwidth}
\textbf{Symptoms}
\begin{itemize}
  \item Too small: loss barely moves.
  \item Just right: loss drops smoothly to a plateau.
  \item Too large: loss oscillates, may even diverge.
\end{itemize}
\column{0.5\textwidth}
\textbf{Practical approach}
\begin{itemize}
  \item Start at $\text{lr} = 10^{-3}$ for Adam.
  \item If loss explodes, reduce by 10$\times$.
  \item If loss plateaus too high, increase by 3$\times$.
  \item Optionally use a scheduler.
\end{itemize}
\end{columns}
\end{frame}

\section*{4. Training discipline}

\begin{frame}{Train / validation / test -- and why all three}
\centering
\begin{tikzpicture}[node distance=0.4cm and 0.6cm, every node/.style={font=\scriptsize}]
\tikzset{box/.style={draw, rounded corners, align=center, minimum height=0.8cm}}
\node[box, fill=blue!10, minimum width=3.6cm] (tr) {Training (60--80\%)\\\emph{model fits these}};
\node[box, fill=orange!12, minimum width=2.0cm, right=of tr] (vl) {Validation (10--20\%)\\\emph{tune hyperparams}};
\node[box, fill=red!12, minimum width=1.4cm, right=of vl] (te) {Test (10--20\%)\\\emph{touch once}};
\end{tikzpicture}
\vspace{0.5em}
\begin{block}{The cardinal rule}
The \textbf{test set is touched once}, at the very end of the project, to estimate generalization. \emph{Every} hyperparameter (learning rate, hidden width, regularizer strength, decision threshold) is chosen using validation, not test.
\end{block}
\begin{alertblock}{Why validation is separate from test}
If you tune on the test set, the test number becomes meaningless -- you have implicitly learned from it. The project uses \emph{group} cross-validation (by scenario and by contingency), not random splits, to avoid leakage between similar samples.
\end{alertblock}
\end{frame}

\begin{block}{The inflection point matters}
Training loss keeps dropping forever; validation loss eventually starts \emph{rising}. The point where validation bottoms out is the best generalizing model. Early stopping = save the model checkpoint at that minimum.
\end{block}

% ============================================================
\begin{frame}{The regularization toolbox}
\centering
\small
\begin{tabular}{lll}
\toprule
Technique & Mechanism & When to reach for it \\
\midrule
\textbf{L2 weight decay} & add $\lambda \|\theta\|^2$ to the loss & always -- almost free \\
\textbf{Dropout} & randomly zero out activations during training & medium / large nets \\
\textbf{Early stopping} & stop at the val-loss minimum & always -- almost free \\
\textbf{Data augmentation} & synthetic variations of training data & images, time series \\
\textbf{Smaller model} & reduce width or depth & quick and effective \\
\bottomrule
\end{tabular}
\vspace{0.5em}
\begin{block}{For the project}
\textbf{L2 weight decay + early stopping is enough.} The dataset is small enough that dropout often hurts (further reducing effective capacity). Week 5/6 notebooks use $\lambda \approx 10^{-4}$ and a patience of 30 epochs.
\end{block}
\end{frame}

% ============================================================
\begin{frame}{Class imbalance and the right metric}
The project's labeled dataset: \textbf{47 unsafe, 30 safe} (61\% positive).
\vspace{0.3em}
\begin{block}{Accuracy is the wrong metric}
A model that predicts ``always unsafe'' achieves 61\% accuracy on training set. A model that's right on 9 out of 10 unsafe but wrong on every safe one achieves $\sim 75\%$ accuracy. Neither is useful.
\end{block}
\vspace{0.2em}
\begin{columns}[T]
\column{0.5\textwidth}
\textbf{Right metrics for safety screening}
\begin{itemize}
  \item \textbf{Recall} on unsafe class
  \item \textbf{FNR}: 1 -- recall (lower is better)
  \item \textbf{AUPRC}: area under precision-recall curve
  \item \textbf{Top-$k$ hit rate}: of the $k$ most-suspicious predictions, how many are truly unsafe?
\end{itemize}
\column{0.5\textwidth}
\textbf{Techniques}
\begin{itemize}
  \item Class-weighted BCE
  \item Decision-threshold tuning (lower threshold favours recall)
  \item Focal loss (advanced; usually unnecessary at our scale)
\end{itemize}
\end{columns}
\end{frame}

\section*{5. A minimal PyTorch example}

\begin{frame}[fragile]{Step 1: data and model}
\begin{lstlisting}[language=Python]
import torch
import torch.nn as nn
from sklearn.datasets import make_moons
from sklearn.model_selection import train_test_split

# Toy data: nonlinearly separable 2D binary classification
X, y = make_moons(n_samples=400, noise=0.20, random_state=0)
X_tr, X_va, y_tr, y_va = train_test_split(X, y, test_size=0.25, random_state=0)
X_tr = torch.tensor(X_tr, dtype=torch.float32)
y_tr = torch.tensor(y_tr, dtype=torch.float32).unsqueeze(1)
X_va = torch.tensor(X_va, dtype=torch.float32)
y_va = torch.tensor(y_va, dtype=torch.float32).unsqueeze(1)

# 2 -> 32 -> 32 -> 1 MLP with sigmoid output
model = nn.Sequential(
    nn.Linear(2, 32),  nn.ReLU(),
    nn.Linear(32, 32), nn.ReLU(),
    nn.Linear(32, 1),  nn.Sigmoid(),
)
\end{lstlisting}
\end{frame}

% ============================================================
\begin{frame}[fragile]{Step 2: training loop}
\begin{lstlisting}[language=Python]
opt   = torch.optim.Adam(model.parameters(), lr=1e-3, weight_decay=1e-4)
bce   = nn.BCELoss()

best_val = float("inf")
for epoch in range(200):
    # --- train ---
    model.train()
    opt.zero_grad()
    loss = bce(model(X_tr), y_tr)
    loss.backward()
    opt.step()

    # --- validate ---
    model.eval()
    with torch.no_grad():
        val_loss = bce(model(X_va), y_va).item()

    if val_loss < best_val:
        best_val = val_loss
        torch.save(model.state_dict(), "best.pt")  # early-stopping checkpoint
\end{lstlisting}
\begin{block}{What this does}
One forward pass on the whole training set per epoch (full-batch, fine for $n=300$), Adam update, validation check, save best model.
\end{block}
\end{frame}

% ============================================================
\begin{frame}[fragile]{Step 3: evaluate and pick the decision threshold}
\begin{lstlisting}[language=Python]
import numpy as np
from sklearn.metrics import confusion_matrix, precision_recall_curve, auc

model.load_state_dict(torch.load("best.pt"))
model.eval()
with torch.no_grad():
    p_va = model(X_va).squeeze().numpy()

# Threshold to maximize recall while keeping precision >= 0.85.
# Note: precision_recall_curve returns len(thr) = len(prec) - 1, so we align with prec[:-1].
prec, rec, thr = precision_recall_curve(y_va.squeeze().numpy(), p_va)
valid = prec[:-1] >= 0.85                     # aligned with thr
threshold = thr[valid][np.argmax(rec[:-1][valid])] if valid.any() else 0.5

y_pred = (p_va >= threshold).astype(int)
tn, fp, fn, tp = confusion_matrix(y_va.squeeze().numpy(), y_pred).ravel()
print(f"recall={tp/(tp+fn):.3f}  precision={tp/(tp+fp):.3f}  AUPRC={auc(rec, prec):.3f}")
\end{lstlisting}
\begin{alertblock}{Decision threshold is a hyperparameter}
Choose it on the validation set, never on test. For screening: prefer thresholds that keep recall high (low FNR), even if precision drops -- false alarms are cheaper than missed violations.
\end{alertblock}
\end{frame}

\section*{6. Pitfalls and the project connection}

\begin{frame}{Common pitfalls in MLP training}
\begin{enumerate}
  \item \warnitem \textbf{Inputs not normalized.} Features with very different scales (e.g., voltage in [0.9, 1.1] vs power in [0, 100]) make optimization slow. Always standardize.
  \item \warnitem \textbf{Wrong output activation.} Sigmoid before BCELoss, but \emph{not} before \code{BCEWithLogitsLoss}. Mixing them up either breaks training or hides bugs.
  \item \warnitem \textbf{Data leakage} between train and test. Feature engineering on the full dataset, then splitting, leaks information. Fit scalers / encoders on train only.
  \item \warnitem \textbf{Tuning on the test set.} You will overfit to it and report optimistic numbers.
  \item \warnitem \textbf{Reporting accuracy on imbalanced data.} The constant predictor often gets a high accuracy. Report recall / FNR / AUPRC.
  \item \warnitem \textbf{Forgetting to seed.} Set \code{torch.manual\_seed(0)} and the numpy and sklearn seeds to make experiments reproducible.
\end{enumerate}
\end{frame}
